# PLAN ‚Äî GUI executable release (bun --compile)

Status: **for review, not started.** Covers packaging the existing local GUI (`src/server.ts` +
`src/gui/*`) as a distributable executable.

Goal: ship the local GUI as a **click-to-run executable** distributed via a GitHub Release,
so an end user needs **no Node, no npm, no source checkout** ‚Äî download one file, double-click,
the browser opens to the GUI.

## Locked decisions

| Decision | Choice | Note |
|---|---|---|
| Packaging tool | **bun `--compile`** | builds a standalone binary that embeds the Bun runtime; **no npm dependency added** (bun is a build-time tool only) |
| First platform | **win-x64** | no code signing needed (SmartScreen shows a one-time "unknown publisher", user clicks through) |
| Other platforms | **deferred** | bun cross-compiles all targets from this one Linux container; adding a platform ‚âà one extra `--target`. See *Deferred*. |
| Distribution | **GitHub Release** with the binary as an asset | git tag ‚Üí release ‚Üí upload `.exe` |
| Config/env resolution | **unchanged** | `flags ‚Üí config.json ‚Üí env` stays (humans + the future AI build share it). See [config.ts](../src/config.ts). |

## Invariants (do not break)

- **Dev workflow unchanged.** `npm run gui` (tsx) keeps serving `src/gui/*` **from disk** so the
  restyle workflow (edit `styles.css`, reload) still works. This honors the frontend/backend
  split frozen in [plan.md](plan.md). Only the **bundled release** inlines the GUI assets ‚Äî the
  embedding must not regress disk-serving in dev.
- **Never bundle secrets.** The artifact must contain **only** the compiled binary ‚Äî never
  `config.json`, `.env`, or `out/`. The on-disk `config.json` holds a real token (gitignored);
  the build script must not pick it up. The end user fills token + DB in the GUI on first run.
- **Engine + pure modules untouched.** Only `server.ts`, `config.ts`, and new build scaffolding
  change. No business logic moves.

## Code changes (two files)

### C1 ‚Äî Embed GUI assets, dual-mode

Today [server.ts:264-278](../src/server.ts#L264) reads `src/gui/{index.html,styles.css,app.js}`
from `GUI_DIR` ([server.ts:87](../src/server.ts#L87)) on every request. A single-file exe has no
such directory, so the assets must travel **inside** the binary.

Approach (bundler-agnostic, keeps dev on disk):

1. `scripts/embed-gui.ts` ‚Äî codegen: read the three GUI files, emit
   `src/gui/embedded.generated.ts` exporting `export const EMBEDDED: Record<string,string>`
   (filename ‚Üí file contents). Gitignored; regenerated by a `prebuild` step.
2. `server.ts` ‚Äî serve through one helper `getAsset(file)`:
   - dev (`GUI_DEV=1`, set by `npm run gui`) ‚Üí `readFile(join(GUI_DIR, file))` (current path,
     hot reload preserved);
   - packaged / default ‚Üí return `EMBEDDED[file]` from memory.
3. Keep the `ROUTES` whitelist and `contentTypeFor` exactly as-is ‚Äî only the **byte source**
   behind a route changes.

> This deliberately re-introduces the "embed as string" idea the spec floated and `plan.md`
> superseded ‚Äî but **scoped to the release build only**, so the dev split survives.

### C2 ‚Äî config/.env path next to the executable

[config.ts:16-17](../src/config.ts#L16) resolves `config.json` / `.env` from `process.cwd()`.
For a double-clicked exe the cwd is unreliable (on macOS it is `/`; on Windows it is usually the
exe folder). When running as a packaged binary, resolve both relative to
`dirname(process.execPath)` instead of `cwd()`, so `config.json` and `out/` land **beside the
exe** predictably.

- Detect "packaged" via a compile-time `--define` (e.g. `IS_PACKAGED`) set only in the bun build;
  unset in dev/CLI/tsx, so CLI behavior (cwd-relative) is unchanged.
- This matters most for macOS later, but wire it now so adding mac needs no second pass here.

## Build scaffolding (new)

`package.json` scripts (no new npm deps ‚Äî bun is external):

```jsonc
"embed:gui":   "tsx scripts/embed-gui.ts",
"prebuild":    "npm run embed:gui",
"build:win":   "bun build --compile --target=bun-windows-x64 --define IS_PACKAGED=true src/server.ts --outfile dist/notion-md-archive.exe"
// deferred ‚Äî uncomment when adding platforms:
// "build:linux": "bun build --compile --target=bun-linux-x64    --define IS_PACKAGED=true src/server.ts --outfile dist/notion-md-archive-linux"
// "build:mac":   "bun build --compile --target=bun-darwin-arm64 --define IS_PACKAGED=true src/server.ts --outfile dist/notion-md-archive-macos-arm64"
```

Prerequisite: **bun installed** in the build environment (the dev container is Node-only today;
`curl -fsSL https://bun.sh/install | bash`, or add it to `postCreateCommand`).

End-user flow: download `.exe` ‚Üí double-click ‚Üí terminal/window starts the server ‚Üí default

## Smoke test (R3, on a real Windows box)

1. Copy only `notion-md-archive.exe` to an empty folder (no Node installed).
2. Double-click ‚Üí browser opens `localhost:4517`.
3. Enter token, Connect ‚Üí database list loads (proves `node:https`/SDK work under bun).
4. Run a `--dry-run` export ‚Üí log streams, `done` event fires (proves SSE + spawn).
5. Confirm `config.json` was written **in that folder**, not elsewhere (proves C2).

## Risks

- **bun node-compat.** The server uses `node:http`, `node:child_process` (`spawn` for the browser
  opener), `node:fs`, `node:https` (SDK). Bun implements these, but verify on Windows in R3 ‚Äî the
  `spawn("start", ‚Ä¶, { shell: true })` browser opener is the most likely wrinkle.
- **Unsigned binary.** Windows SmartScreen warns once ("unknown publisher"); acceptable for v1.
  macOS Gatekeeper is stricter (needs `xattr -d` or a $99/yr Apple signing cert) ‚Äî a reason mac
  is deferred.
- **No hot-reload in the exe** (assets are frozen at build). Intentional; dev keeps disk-serve.

## Deferred (cheap to add later)

- **linux-x64:** uncomment `build:linux`; tell users `chmod +x`. (glibc build; only very old
  distros would need a `-musl` target.)
- **macOS (arm64 + x64):** uncomment targets ‚Äî builds fine ‚Äî but a clean install needs code
  signing + notarization ($99/yr Apple Developer). Two binaries (Apple Silicon + Intel).
- **CI** (R5) so tag push builds every target without a local machine per OS.
